# User Stories & Use Cases — Omnichannel Messenger V1.1

> Полный набор user stories для SaaS unified inbox (Telegram, Slack, WhatsApp, Gmail).
> Формат: Эпик → User Story → Acceptance Criteria.
> Роли: **Superadmin** (SA), **Admin** (A), **User** (U).

---

## Эпик 1: Аутентификация и Онбординг

### US-1.1: Регистрация по приглашению
**Как** приглашённый пользователь, **я хочу** зарегистрироваться по ссылке из email, **чтобы** получить доступ к платформе.

**Acceptance Criteria:**
- [ ] При переходе по ссылке с валидным токеном открывается форма регистрации (имя, пароль)
- [ ] После заполнения формы создаётся аккаунт со статусом `active` и назначенной ролью
- [ ] Пользователь автоматически авторизуется и попадает на Dashboard
- [ ] При переходе по истёкшему токену (>7 дней) показывается ошибка "Ссылка устарела"
- [ ] При повторном использовании токена показывается ошибка "Приглашение уже использовано"
- [ ] Пароль проходит валидацию: минимум 8 символов, буквы + цифры
- [ ] В Activity Log создаётся запись `user_created`

### US-1.2: Вход в систему
**Как** зарегистрированный пользователь, **я хочу** войти по email и паролю, **чтобы** получить доступ к своей организации.

**Acceptance Criteria:**
- [ ] При верных credentials выдаётся JWT access token (15 мин) и refresh token (httpOnly cookie, 7 дней)
- [ ] Пользователь перенаправляется на Dashboard
- [ ] При неверном пароле показывается ошибка "Неверный email или пароль" (без уточнения, что именно неверно)
- [ ] После 5 неудачных попыток за минуту аккаунт временно блокируется (rate limit)
- [ ] Деактивированный пользователь (`status=deactivated`) получает ошибку "Аккаунт деактивирован"
- [ ] Пользователь из suspended организации получает ошибку "Организация приостановлена"

### US-1.3: Автоматическое обновление токена
**Как** авторизованный пользователь, **я хочу**, чтобы сессия обновлялась автоматически, **чтобы** не вводить пароль каждые 15 минут.

**Acceptance Criteria:**
- [ ] При получении 401 фронтенд автоматически отправляет refresh token на `/api/auth/refresh`
- [ ] При успешном обновлении оригинальный запрос повторяется прозрачно для пользователя
- [ ] При невалидном refresh token пользователь перенаправляется на страницу входа
- [ ] Старый refresh token инвалидируется после использования (rotation)
- [ ] При параллельных запросах только один обновляет токен, остальные ждут

### US-1.4: Выход из системы
**Как** пользователь, **я хочу** выйти из аккаунта, **чтобы** защитить свои данные на общем устройстве.

**Acceptance Criteria:**
- [ ] При нажатии "Выход" refresh token удаляется из БД и cookie очищается
- [ ] WebSocket соединение закрывается
- [ ] Пользователь перенаправляется на страницу входа
- [ ] Кэш React Query очищается (нет данных предыдущей сессии)

### US-1.5: Смена пароля
**Как** пользователь, **я хочу** сменить пароль в настройках профиля, **чтобы** обеспечить безопасность аккаунта.

**Acceptance Criteria:**
- [ ] Форма требует ввести текущий пароль и новый пароль (дважды)
- [ ] При неверном текущем пароле показывается ошибка
- [ ] Новый пароль должен отличаться от текущего
- [ ] После смены все активные refresh tokens инвалидируются (кроме текущего)
- [ ] Показывается уведомление "Пароль успешно изменён"

### US-1.6: Восстановление пароля
**Как** пользователь, забывший пароль, **я хочу** восстановить доступ через email, **чтобы** не потерять аккаунт.

> **Примечание:** Требуется добавить в Technical Spec: `POST /api/auth/forgot-password`, `POST /api/auth/reset-password` и модель `PasswordResetToken` в схему Prisma.

**Acceptance Criteria:**
- [ ] На странице входа есть ссылка "Забыли пароль?"
- [ ] `POST /api/auth/forgot-password` принимает email, генерирует токен (JWT, 1 час), сохраняет в `PasswordResetToken`
- [ ] На email отправляется письмо с ссылкой для сброса
- [ ] `POST /api/auth/reset-password` принимает токен + новый пароль, валидирует и обновляет
- [ ] После сброса все refresh tokens инвалидируются
- [ ] При вводе несуществующего email показывается такое же сообщение (защита от перебора)
- [ ] Повторный запрос в течение 1 минуты блокируется (rate limit)
- [ ] При неудачной отправке email — ошибка логируется, пользователю показывается общее сообщение

### US-1.7: Перенаправление неавторизованных
**Как** система, **я должна** перенаправлять неавторизованных пользователей на страницу входа, **чтобы** защитить данные.

**Acceptance Criteria:**
- [ ] Доступ к любому route кроме `/login`, `/register`, `/reset-password` без токена → редирект на `/login`
- [ ] После успешного входа пользователь возвращается на URL, который он изначально запрашивал
- [ ] При потере сетевого соединения показывается баннер "Соединение потеряно" с автоматическим реконнектом

### US-1.8: Валидация формы регистрации
**Как** пользователь, **я хочу** видеть ошибки валидации в реальном времени, **чтобы** быстро заполнить форму.

**Acceptance Criteria:**
- [ ] Имя: обязательное поле, 2-100 символов
- [ ] Пароль: минимум 8 символов, показывается индикатор надёжности
- [ ] Подтверждение пароля: совпадает с паролем
- [ ] Ошибки показываются при потере фокуса поля (onBlur)
- [ ] Кнопка "Зарегистрироваться" неактивна пока форма невалидна

### US-1.9: Множество вкладок браузера
**Как** пользователь, **я хочу** работать в нескольких вкладках одновременно, **чтобы** открывать разные чаты параллельно.

**Acceptance Criteria:**
- [ ] Каждая вкладка имеет своё WebSocket соединение
- [ ] Обновление токена в одной вкладке синхронизируется с другими через BroadcastChannel API (т.к. refresh token rotation инвалидирует старый)
- [ ] Выход из одной вкладки разлогинивает все вкладки (через BroadcastChannel)

### US-1.10: Онбординг нового пользователя
**Как** новый пользователь после регистрации, **я хочу** понять первые шаги, **чтобы** быстро начать работу.

**Acceptance Criteria:**
- [ ] После первого входа показывается welcome-экран с шагами: 1) Подключите мессенджер 2) Импортируйте чаты 3) Отправьте первое сообщение
- [ ] Каждый шаг — это ссылка на соответствующую страницу (Настройки → Интеграции, Чаты → + Добавить чат)
- [ ] Пользователь может пропустить онбординг ("Пропустить")
- [ ] Онбординг не показывается при повторных входах

## Эпик 2: Управление Организациями

### US-2.1: Просмотр списка организаций
**Как** Superadmin, **я хочу** видеть все организации на платформе, **чтобы** управлять ими.

**Acceptance Criteria:**
- [ ] Таблица: Название, Кол-во пользователей, Дата создания, Статус (Active/Suspended)
- [ ] Поиск по названию организации
- [ ] Сортировка по каждому столбцу
- [ ] Доступ только для роли Superadmin — Admin/User получают 403

### US-2.2: Создание организации
**Как** Superadmin, **я хочу** создать новую организацию, **чтобы** подключить нового клиента.

**Acceptance Criteria:**
- [ ] Форма: название организации, email администратора
- [ ] После создания организация получает статус `active`
- [ ] Администратору отправляется приглашение по email
- [ ] В Activity Log создаётся запись `organization_created`
- [ ] Название организации уникально в системе

### US-2.3: Редактирование организации
**Как** Superadmin, **я хочу** изменить данные организации, **чтобы** актуализировать информацию.

**Acceptance Criteria:**
- [ ] Можно изменить: название, логотип, язык по умолчанию, часовой пояс
- [ ] Изменения сохраняются мгновенно
- [ ] В Activity Log создаётся запись `organization_settings_changed`

### US-2.4: Приостановка организации
**Как** Superadmin, **я хочу** приостановить организацию, **чтобы** заблокировать доступ при неоплате.

**Acceptance Criteria:**
- [ ] При нажатии "Приостановить" → подтверждение → статус меняется на `suspended`
- [ ] Все пользователи организации автоматически разлогиниваются
- [ ] Пользователи видят "Организация приостановлена" при попытке входа
- [ ] Рассылки останавливаются, новые сообщения не отправляются
- [ ] Входящие сообщения продолжают сохраняться в БД
- [ ] В Activity Log: `organization_suspended`

### US-2.5: Активация организации
**Как** Superadmin, **я хочу** реактивировать организацию, **чтобы** восстановить доступ клиенту.

**Acceptance Criteria:**
- [ ] Статус меняется на `active`
- [ ] Пользователи могут снова войти
- [ ] Все интеграции возобновляют работу (если токены не истекли)
- [ ] В Activity Log: `organization_activated`

### US-2.6: Глобальные лимиты рассылок
**Как** Superadmin, **я хочу** установить максимальные лимиты рассылок для организации, **чтобы** предотвратить злоупотребления.

**Acceptance Criteria:**
- [ ] Настройки: макс. сообщений/день, макс. рассылок/день на организацию
- [ ] Админы организации не могут превысить эти лимиты в своих антибан-настройках
- [ ] При достижении лимита рассылка ставится в очередь на следующий день
- [ ] В Activity Log: `organization_limits_changed`

### US-2.7: Удаление организации
**Как** Superadmin, **я хочу** удалить организацию, **чтобы** полностью очистить данные клиента.

**Acceptance Criteria:**
- [ ] Требуется подтверждение с вводом названия организации
- [ ] Каскадное удаление: пользователи, чаты, сообщения, рассылки, интеграции
- [ ] Действие необратимо — показывается предупреждение
- [ ] В Activity Log (глобальном): `organization_deleted`

### US-2.8: Переключение между организациями (Superadmin)
**Как** Superadmin, **я хочу** переключаться между организациями в UI, **чтобы** управлять разными клиентами.

**Acceptance Criteria:**
- [ ] В header или sidebar — dropdown/switcher со списком организаций
- [ ] При переключении обновляются: чаты, пользователи, рассылки, настройки
- [ ] WebSocket подписки переключаются на новую организацию
- [ ] Текущая организация отображается в UI (название/логотип)

### US-2.9: Изоляция данных между организациями
**Как** Admin организации A, **я не должен** видеть данные организации B, **чтобы** обеспечить безопасность.

**Acceptance Criteria:**
- [ ] Все API-запросы фильтруются по `organizationId` из JWT
- [ ] Попытка доступа к чату/пользователю другой организации возвращает 404 (не 403, чтобы не раскрывать существование)
- [ ] WebSocket события доставляются только пользователям своей организации
- [ ] Superadmin может переключаться между организациями

## Эпик 3: Управление Пользователями

### US-3.1: Приглашение пользователя
**Как** Admin, **я хочу** пригласить нового пользователя в организацию, **чтобы** расширить команду.

**Acceptance Criteria:**
- [ ] Форма: email, роль (Admin/User)
- [ ] На email отправляется приглашение со ссылкой для регистрации
- [ ] Нельзя пригласить email, который уже есть в организации
- [ ] Можно повторно отправить приглашение (re-send)
- [ ] User не имеет доступа к приглашению → 403
- [ ] В Activity Log: `user_invited`

### US-3.2: Просмотр списка пользователей
**Как** Admin, **я хочу** видеть всех пользователей организации, **чтобы** управлять командой.

**Acceptance Criteria:**
- [ ] Таблица: Имя, Email, Роль, Статус, Последняя активность, Кол-во импортированных чатов
- [ ] Поиск по имени и email
- [ ] Фильтр по роли и статусу
- [ ] User видит только свой профиль (или всех, если Admin разрешил)
- [ ] Superadmin видит пользователей всех организаций

### US-3.3: Изменение роли пользователя
**Как** Admin, **я хочу** изменить роль пользователя, **чтобы** настроить уровень доступа.

**Acceptance Criteria:**
- [ ] Admin может повысить User до Admin или понизить Admin до User
- [ ] Нельзя понизить самого себя (последний Admin организации)
- [ ] Изменения применяются мгновенно — при следующем запросе проверяется новая роль
- [ ] В Activity Log: `user_role_changed`

### US-3.4: Деактивация пользователя
**Как** Admin, **я хочу** деактивировать пользователя, **чтобы** заблокировать его доступ без удаления данных.

**Acceptance Criteria:**
- [ ] Статус меняется на `deactivated`
- [ ] Пользователь автоматически разлогинивается
- [ ] Нельзя войти с деактивированным аккаунтом
- [ ] Чаты, где он Owner, сохраняются (но можно переназначить)
- [ ] Нельзя деактивировать самого себя
- [ ] В Activity Log: `user_deactivated`

### US-3.5: Реактивация пользователя
**Как** Admin, **я хочу** реактивировать пользователя, **чтобы** вернуть ему доступ.

**Acceptance Criteria:**
- [ ] Статус меняется на `active`
- [ ] Пользователь может снова войти с прежним паролем
- [ ] Все его данные (чаты, сообщения) сохранены
- [ ] В Activity Log: `user_reactivated`

### US-3.6: Редактирование профиля
**Как** пользователь, **я хочу** обновить свой профиль, **чтобы** коллеги видели актуальную информацию.

**Acceptance Criteria:**
- [ ] Можно изменить: имя, аватар
- [ ] Email отображается как read-only
- [ ] Аватар: загрузка изображения (макс. 5MB, JPEG/PNG)
- [ ] Изменения отражаются везде (сообщения, списки пользователей)

### US-3.7: Настройка видимости чатов
**Как** Admin, **я хочу** управлять видимостью чатов для обычных пользователей, **чтобы** контролировать доступ к информации.

**Acceptance Criteria:**
- [ ] Чекбокс "Пользователи видят все чаты организации" в Настройки → Рабочее пространство
- [ ] Включён: User видит все чаты в "Все чаты"
- [ ] Выключен: User видит только чаты, где он Owner или Imported By
- [ ] "Мои чаты" всегда показывает только собственные чаты
- [ ] Изменение применяется мгновенно для всех пользователей
- [ ] В Activity Log: `chat_visibility_changed`

### US-3.8: Суперадмин создаёт админа другой организации
**Как** Superadmin, **я хочу** назначить администратора в любой организации, **чтобы** делегировать управление.

**Acceptance Criteria:**
- [ ] Superadmin может приглашать пользователей в любую организацию
- [ ] Superadmin может менять роли пользователей в любой организации
- [ ] Действия логируются с указанием организации

### US-3.9: Ограничения роли User
**Как** система, **я должна** ограничить права роли User, **чтобы** защитить настройки организации.

**Acceptance Criteria:**
- [ ] User НЕ может: приглашать пользователей, менять роли, настраивать интеграции, редактировать антибан, управлять тегами, менять workspace-настройки
- [ ] User МОЖЕТ: отправлять сообщения, импортировать чаты, использовать шаблоны, просматривать свои рассылки
- [ ] При попытке доступа к запрещённым функциям → 403 + соответствующий элемент UI скрыт/задизейблен

## Эпик 4: Интеграции с Мессенджерами

### US-4.1: Просмотр интеграций
**Как** пользователь, **я хочу** видеть статус подключений всех мессенджеров, **чтобы** понимать, какие каналы доступны.

**Acceptance Criteria:**
- [ ] Экран показывает 4 карточки: Telegram, Slack, WhatsApp, Gmail
- [ ] Каждая карточка: иконка мессенджера, статус (Connected/Disconnected/Token Expired/Session Expired)
- [ ] Для подключённых — кнопки "Отключить", "Настройки"
- [ ] Для отключённых — кнопка "Подключить"
- [ ] User видит карточки в режиме просмотра (не может подключать/отключать)

### US-4.2: Подключение Telegram
**Как** Admin, **я хочу** подключить Telegram-аккаунт, **чтобы** отправлять и получать сообщения из Telegram-чатов.

**Acceptance Criteria:**
- [ ] Вход по API token (API ID + API Hash) или через MTProto-сессию
- [ ] Многошаговая авторизация: ввод номера телефона → код из Telegram → (опционально) 2FA пароль
- [ ] После подключения статус меняется на `connected`
- [ ] Учётные данные (session) шифруются перед сохранением в БД (AES-256-GCM)
- [ ] При ошибке авторизации показывается понятное сообщение
- [ ] В Activity Log: `integration_connected` (messenger: telegram)

### US-4.3: Подключение Slack
**Как** Admin, **я хочу** подключить Slack workspace, **чтобы** управлять сообщениями в каналах и DM.

**Acceptance Criteria:**
- [ ] OAuth flow: кнопка "Add to Slack" → авторизация в Slack → callback → токен сохранён
- [ ] Запрашиваемые scopes: channels:read, channels:write, chat:write, users:read, etc.
- [ ] После подключения статус `connected`, отображается имя workspace
- [ ] Токен шифруется перед сохранением
- [ ] При отклонении авторизации пользователем → возврат на страницу интеграций с ошибкой
- [ ] В Activity Log: `integration_connected` (messenger: slack)

### US-4.4: Подключение WhatsApp
**Как** Admin, **я хочу** подключить WhatsApp, **чтобы** отправлять сообщения через WhatsApp.

**Acceptance Criteria:**
- [ ] Подключение через QR-код (Baileys) или Business API
- [ ] QR-код отображается в модальном окне, обновляется каждые 30 секунд
- [ ] После сканирования QR-кода сессия сохраняется (зашифрованная)
- [ ] Статус `connected` после успешного подключения
- [ ] Если сессия истекает → статус `session_expired`, предложение переподключиться
- [ ] В Activity Log: `integration_connected` (messenger: whatsapp)

### US-4.5: Подключение Gmail
**Как** Admin, **я хочу** подключить Gmail-аккаунт, **чтобы** управлять email-перепиской через платформу.

**Acceptance Criteria:**
- [ ] Google OAuth flow: кнопка → Google consent screen → callback → tokens сохранены
- [ ] Scopes: gmail.readonly, gmail.send, gmail.modify
- [ ] После подключения отображается email-адрес аккаунта
- [ ] Refresh token сохраняется зашифрованным, access token обновляется автоматически
- [ ] В Activity Log: `integration_connected` (messenger: gmail)

### US-4.6: Отключение интеграции
**Как** Admin, **я хочу** отключить мессенджер, **чтобы** прекратить синхронизацию.

**Acceptance Criteria:**
- [ ] Подтверждение: "Все чаты этого мессенджера станут read-only. Продолжить?"
- [ ] Статус интеграции → `disconnected`

- [ ] При чтении credentials расшифровываются на лету
- [ ] В БД хранится base64 строка (не plaintext JSON)

### US-4.11: Ограничения роли User для интеграций

### US-4.13: Индикация состояния интеграции в UI

- [ ] WhatsApp: ограниченная поддержка (UI only)

### US-6.15: Форматирование текста по мессенджерам
**Как** пользователь, **я хочу** использовать форматирование в сообщениях, **чтобы** текст корректно отображался в каждом мессенджере.

- [ ] Delivered (✓✓): доставлено получателю
- [ ] Read (✓✓ синие): прочитано (поддерживается Slack/Gmail)

### US-6.19: Добавление в избранное

## Эпик 7: Рассылки (Broadcasts)

### US-7.1: Просмотр списка рассылок
**Как** пользователь, **я хочу** видеть все рассылки организации, **чтобы** отслеживать их статус.

**Acceptance Criteria:**
- [ ] Таблица: Название, Статус (Draft/Scheduled/Sending/Sent/Partially Failed/Failed), Чаты (кол-во), Мессенджеры, Создал, Отправлено, Delivery rate %, Failed (кол-во)
- [ ] Статусы цветокодированы (Draft=серый, Sending=синий, Sent=зелёный, Failed=красный)
- [ ] Клик по строке → детали рассылки
- [ ] User видит только свои рассылки; Admin/Superadmin — все в организации

### US-7.2: Визард создания рассылки — Шаг 1: Контент
**Как** пользователь, **я хочу** составить текст рассылки, **чтобы** отправить его в выбранные чаты.

**Acceptance Criteria:**
- [ ] Поле "Название рассылки" (обязательное)
- [ ] Textarea "Текст сообщения" с поддержкой Markdown и live-preview
- [ ] Кнопка "Использовать шаблон" → dropdown с шаблонами → текст подставляется
- [ ] Кнопка "Прикрепить файл" (выбор файлов, загрузка)
- [ ] Нельзя перейти к шагу 2 без текста сообщения
- [ ] Переменные: `{{chat_name}}`, `{{owner_name}}`, `{{date}}` — подсвечиваются в preview

### US-7.3: Визард — Шаг 2: Получатели
**Как** пользователь, **я хочу** выбрать чаты для рассылки, **чтобы** определить аудиторию.

**Acceptance Criteria:**
- [ ] Мультивыбор из списка чатов организации
- [ ] Фильтры: по мессенджеру, по тегам, по владельцу
- [ ] Read-only чаты недоступны для выбора (серые + подсказка "Мессенджер отключён")
- [ ] Отображение: "Выбрано X чатов" (с разбивкой по мессенджерам)
- [ ] Нельзя перейти к шагу 3 без хотя бы одного чата

### US-7.4: Визард — Шаг 3: Расписание
**Как** пользователь, **я хочу** отправить рассылку сейчас или запланировать, **чтобы** контролировать время отправки.

**Acceptance Criteria:**
- [ ] Radio-кнопки: "Отправить сейчас" / "Запланировать"
- [ ] При выборе "Запланировать" → date-time picker
- [ ] Нельзя выбрать время в прошлом
- [ ] Часовой пояс: берётся из настроек организации

### US-7.5: Визард — Шаг 4: Обзор и подтверждение
**Как** пользователь, **я хочу** проверить все параметры перед отправкой, **чтобы** избежать ошибок.

**Acceptance Criteria:**
- [ ] Суммарная информация: название, превью текста, кол-во чатов, мессенджеры
- [ ] Risk Indicator — цветовой индикатор на основе антибан-настроек
- [ ] Оценочное время отправки всех сообщений
- [ ] Кнопки: "Отправить" / "Запланировать" / "Назад" / "Сохранить как черновик"
- [ ] При высоком риске (Red zone) → предупреждение с подтверждением

### US-7.6: Отправка рассылки с антибан-ограничениями
**Как** система, **я должна** отправлять сообщения с задержками, **чтобы** не попасть под бан мессенджеров.

**Acceptance Criteria:**
- [ ] Сообщения отправляются последовательно с задержками из антибан-настроек
- [ ] Для каждого мессенджера свои настройки: batch size, delay between messages, delay between batches
- [ ] Не превышаются лимиты: max messages/hour, max messages/day
- [ ] BullMQ child jobs с вычисленными delayed timestamps
- [ ] Статус обновляется для каждого чата: pending → sent / failed
- [ ] Рассылка от имени пользователя-создателя (не бота)

### US-7.7: Прогресс рассылки в реальном времени
**Как** пользователь, **я хочу** видеть прогресс отправки, **чтобы** понимать текущее состояние.

**Acceptance Criteria:**
- [ ] Progress bar с процентом доставки
- [ ] WebSocket `ws:broadcast_status` с обновлениями
- [ ] Счётчики: отправлено / ошибки / в очереди
- [ ] Обновление каждые 2-5 секунд

### US-7.8: Детали рассылки
**Как** пользователь, **я хочу** видеть детальную статистику рассылки, **чтобы** оценить результат.

**Acceptance Criteria:**
- [ ] Суммарная информация: название, статус, создатель, дата
- [ ] Статистика доставки: всего, доставлено, delivery rate %, ошибки
- [ ] Таблица ошибок: имя чата, мессенджер, причина ошибки, статус retry
- [ ] Кнопка "Повторить для ошибочных" (если есть failed)

### US-7.9: Автоматический повтор ошибочных
**Как** система, **я должна** автоматически повторять неудачные отправки, **чтобы** повысить delivery rate.

**Acceptance Criteria:**
- [ ] Включается в настройках (Settings → Broadcast Settings → Auto-retry toggle)
- [ ] Экспоненциальный backoff: delay × 2^attempt (1x → 2x → 4x)
- [ ] Максимум попыток: настраиваемый (1-5, по умолчанию 3)
- [ ] Окно повтора: 1-24 часа (настраиваемый)
- [ ] После исчерпания попыток: статус `retry_exhausted`
- [ ] В Activity Log: `broadcast_retry_succeeded` или `broadcast_retry_failed`

### US-7.10: Ручной повтор для ошибочных чатов
**Как** пользователь, **я хочу** вручную повторить отправку для ошибочных чатов, **чтобы** довести рассылку до конца.

**Acceptance Criteria:**
- [ ] Кнопка "Повторить для ошибочных" на странице деталей рассылки
- [ ] Повторяет только чаты со статусом `failed` или `retry_exhausted`
- [ ] Использует текущие антибан-настройки
- [ ] В Activity Log: `broadcast_retry_initiated`

### US-7.11: Аналитика рассылок
**Как** Admin, **я хочу** видеть общую аналитику по рассылкам, **чтобы** оценить эффективность.

**Acceptance Criteria:**
- [ ] Глобальные метрики: всего рассылок, отправлено сообщений, delivery rate %, response rate %, среднее время до ответа, open rate (Gmail)
- [ ] Разбивка по мессенджерам: таблица (мессенджер, отправлено, доставлено, %, ошибки, ответы, %, средн. время)
- [ ] Фильтры: период (неделя/месяц/квартал/custom), мессенджер, создатель
- [ ] User видит аналитику только по своим рассылкам

### US-7.12: Редактирование черновика рассылки
**Как** пользователь, **я хочу** редактировать черновик рассылки, **чтобы** доработать его перед отправкой.

**Acceptance Criteria:**
- [ ] Клик по рассылке со статусом `draft` → открывается визард с заполненными данными
- [ ] Можно изменить любой шаг (контент, получатели, расписание)
- [ ] Кнопка "Сохранить" обновляет черновик
- [ ] Нельзя редактировать рассылки со статусом `sending`, `sent`, `failed`

### US-7.13: Дублирование рассылки
**Как** пользователь, **я хочу** дублировать существующую рассылку, **чтобы** быстро создать похожую.

**Acceptance Criteria:**
- [ ] Кнопка "Дублировать" → создаётся новый черновик с теми же данными (контент, получатели)
- [ ] Название: "Копия — [оригинальное название]"
- [ ] Расписание сбрасывается (нужно выбрать заново)
- [ ] Доступно для рассылок любого статуса

### US-7.14: Удаление рассылки
**Как** пользователь, **я хочу** удалить рассылку, **чтобы** убрать ненужные.

**Acceptance Criteria:**
- [ ] Можно удалить только `draft` и `scheduled` рассылки
- [ ] Scheduled рассылка: отменяется запланированная отправка + удаляется
- [ ] Подтверждение перед удалением
- [ ] Sent/Failed рассылки нельзя удалить (только архивировать)

### US-7.15: Восстановление рассылки после сбоя worker
**Как** система, **я должна** корректно возобновлять рассылку после перезапуска worker, **чтобы** не дублировать и не терять сообщения.

> **Важно:** Worker перезапускается при деплоях на Railway. Обработка должна быть идемпотентной.

**Acceptance Criteria:**
- [ ] BullMQ jobs сохраняются в Redis (переживают рестарт worker)
- [ ] При рестарте worker подхватывает незавершённые jobs
- [ ] Каждый job — идемпотентный: повторная обработка не дублирует сообщение (проверка `BroadcastChat.status`)
- [ ] Если job уже отправлен (`status=sent`) — пропускается
- [ ] Broadcast status корректно рассчитывается из суммы BroadcastChat статусов
- [ ] Логируется факт recovery: "Broadcast job resumed after worker restart"

### US-7.16: Реактивная обработка rate-limit ошибок от мессенджеров
**Как** система, **я должна** корректно обрабатывать 429 ответы от API мессенджеров, **чтобы** не усугублять ситуацию.

**Acceptance Criteria:**
- [ ] При получении 429 от Telegram/Slack/WhatsApp/Gmail API — job переставляется с увеличенной задержкой
- [ ] Используется `Retry-After` header из ответа мессенджера (если есть)
- [ ] Если `Retry-After` отсутствует — exponential backoff (30с, 60с, 120с)
- [ ] Оставшиеся jobs в текущем batch тоже задерживаются
- [ ] В Activity Log: `messenger_rate_limited` (messenger, retry_after)
- [ ] В UI рассылки показывается предупреждение "Отправка замедлена из-за ограничений мессенджера"

### US-7.17: Переходы статусов рассылки
**Как** система, **я должна** корректно управлять статусами рассылки, **чтобы** отражать реальное состояние.

**Acceptance Criteria:**
- [ ] Draft → Scheduled (при планировании) или Sending (при немедленной отправке)
- [ ] Scheduled → Sending (когда наступает время)
- [ ] Sending → Sent (все чаты успешны) или Partially Failed (часть не доставлена) или Failed (все ошибки)
- [ ] Partially Failed → можно повторить для ошибочных
- [ ] Каждый переход логируется в Activity Log

### US-13.1: Подключение WebSocket

### US-13.2: Получение сообщений в реальном времени
**Как** пользователь, **я хочу** получать новые сообщения мгновенно, **чтобы** не обновлять страницу.

**Acceptance Criteria:**
- [ ] Событие `ws:new_message` обновляет: ленту сообщений (если чат открыт), превью в списке чатов, счётчик непрочитанных, виджет на дашборде
- [ ] React Query кэш инвалидируется для обновлённых данных
- [ ] Новое сообщение анимированно появляется в ленте

### US-13.3: Обновление сообщений
**Как** система, **я должна** синхронизировать редактирование и удаление сообщений, **чтобы** все видели актуальное состояние.

- [ ] `ws:message_updated` → текст обновляется + метка "(отредактировано)" появляется
- [ ] `ws:message_deleted` → сообщение удаляется из ленты (или заменяется на "Сообщение удалено")

### US-13.4: Индикатор набора текста

- [ ] Индикатор исчезает через 3 секунды без активности

### US-13.7: Прогресс рассылки через WebSocket

### US-13.8: Изоляция каналов по организации

## Сквозные (Cross-cutting) критерии

Следующие критерии применяются ко ВСЕМ user stories:

### CC-1: Мультитенантная изоляция
- [ ] Каждый API-запрос фильтрует по `organizationId` из JWT
- [ ] Попытка доступа к чужим данным → 404 (не 403)

### CC-2: RBAC
- [ ] Каждый эндпоинт проверяет роль через `requireRole()` middleware
- [ ] UI скрывает/дизейблит элементы для недостаточных ролей

### CC-3: Обработка ошибок
- [ ] Формат ошибок: `{ error: { code, message, statusCode } }`
- [ ] 401 → автообновление токена → retry
- [ ] 403 → toast "Недостаточно прав"
- [ ] 422 → показ ошибок валидации у полей
- [ ] 429 → toast "Слишком много запросов, подождите"
- [ ] 500 → toast "Внутренняя ошибка, попробуйте позже"
- [ ] Сетевая ошибка → баннер "Соединение потеряно"

### CC-4: Логирование
- [ ] Каждое значимое действие создаёт запись в Activity Log
- [ ] Запись содержит: userId, organizationId, category, action, targetType, targetId, metadata, timestamp

### CC-5: Безопасность
- [ ] CORS: только `APP_URL`
- [ ] Rate limiting: 100 req/min general, 10 req/min auth
- [ ] Body size limit: 10MB
- [ ] Input validation на каждом эндпоинте (Zod)
- [ ] SQL injection prevention (Prisma)
- [ ] XSS prevention (sanitize user content)

## Открытые вопросы для реализации

1. **Hard vs Soft Delete чатов** — текущая Prisma-схема использует каскадное удаление. Нужно ли добавить `deletedAt` для soft-delete?
2. **Password Reset** — требуется добавить эндпоинты `POST /api/auth/forgot-password` и `POST /api/auth/reset-password` + модель `PasswordResetToken` в схему.
3. **Forward API** — требуется спроектировать `POST /api/messages/:id/forward` с поддержкой кросс-мессенджерной пересылки.
4. **Organization Delete** — требуется `DELETE /api/organizations/:id` в Technical Spec.
5. **`ws:message_pinned`** — событие используется в stories, но отсутствует в Technical Spec WebSocket events list.
